--------------------------------------------------------------------------------
 TOP TECHNICAL DECISIONS (AND WHY)
--------------------------------------------------------------------------------

1. Money as integer minor units, never float/double, in a hand-rolled
   `Money` value object on both sides — no money.js/dinero.js-style
   dependency. For 2 currencies, a ~40-line VO is simpler and carries zero
   dependency risk than pulling in a library. Rounds half-up **exactly
   once**, at the boundary, avoiding double-rounding. Backed by Mexico's
   Monetary Law (5-cent rounding is cash-only; a transfer is non-cash → we
   round to the exact cent) and Banxico's FIX rate precision. The rate is
   never hardcoded on the client — it comes from the backend on every quote.

2. Idempotency in 3 independent layers: a single-use quote, an
   `Idempotency-Key` header (Stripe's pattern), and a Flutter double-submit
   UI guard. The backend never trusts client-sent money — `POST /transfers`
   carries only `quoteId`; the backend recalculates from its own stored,
   HMAC-signed quote, so a tampered body is structurally impossible to act
   on, not just validated away.

3. Hexagonal-light backend (ports for RateProvider/Repository/Clock, no
   per-layer DTOs or a command bus) + feature-first Clean Arch on Flutter
   (domain/data/presentation). Scoped down from a fuller layering a couple
   of times during the build — per-layer DTOs and heavier process
   scaffolding (ADRs, contract-checker) got cut as ceremony with no payoff
   for a 2-endpoint slice (YAGNI over completeness).


--------------------------------------------------------------------------------
 SOMETHING THE AI GOT WRONG, AND HOW IT WAS CAUGHT
--------------------------------------------------------------------------------

The clearest case: two Riverpod notifiers had `late final` fields assigned
inside `build()`. Riverpod's contract allows `build()` to re-run on the same
notifier instance on some invalidation edge cases (here: resetting a
provider from a "Send another" action, in the same frame a new widget
subscribed to it) — and a `late final` throws on the second assignment. That
surfaced as a raw `LateError`, which a UI force-cast (`error as Failure`)
then turned into an app crash. This wasn't caught by a unit test — it only
shows up on a specific navigation sequence — but by manually exercising the
app screen-by-screen after the feature looked "done." Fixed by dropping
`final` on both fields and hardening every force-cast into a safe check that
falls back to a generic error instead of crashing on an unrecognized type.

Two more, caught by the ts-reviewer/flutter-reviewer subagents on the diff
before merge (both compile, pass the happy path, and only break under a race
or malformed input):
- A Flutter datasource mapped the JSON response to the domain type OUTSIDE
  its try/catch, so a malformed 2xx body threw an uncaught exception instead
  of a typed `Err`.
- A debounced quote notifier didn't bump its request-guard id when the input
  was cleared, so a stale in-flight reply for an old amount could overwrite
  the "idle" state after the field was emptied.

Both were fixed with a regression test the same day they were found.


--------------------------------------------------------------------------------
 CORNERS CUT, AND WHAT I'D DO WITH MORE TIME
--------------------------------------------------------------------------------

Cut for this slice (and why that's fine here):
- Public deploy + a signed release APK — infrastructure work with no bearing
  on the code being evaluated; running both sides locally is enough (see
  README.md).
- Certificate pinning — the evaluation window is short and PaaS providers
  rotate certs; TLS + a public CA covers this scope.
- AuthN/AuthZ, KYC/AML — explicitly out of scope per the brief.
- A real database — in-memory behind a Repository port, swappable later
  without touching the domain.
- Client-side min/max amount validation — delegated to the backend, the
  single source of truth for that business rule.

With more time, in priority order:
1. Auth, scoping quotes/transfers to the authenticated user — right now
   anyone holding a `quoteId` could confirm it; this is gap #1 for a real
   launch (documented in security.md).
2. A `contract-checker` to keep the backend↔frontend API contract from
   drifting once it grows past this single pair of endpoints.
3. Real persistence, more currency pairs, and picking the transfer
   direction (today only MXN→USD).


--------------------------------------------------------------------------------
 TIME SPENT
--------------------------------------------------------------------------------

The brief scopes this at 2-3h. My own active time — reviewing each PR,
directing the work, verifying on-device, giving feedback — was **~3-4h**,
close to that scope.

The table below tracks something different: session/build duration per PR,
which includes stretches where the AI executed autonomously between my
check-ins, across multiple sessions spread over more than one day. Summed
up it comes to ~14h40m estimated / ~18h30m of that cumulative time — not
hours I sat continuously working, but still a real signal of where the
build concentrated effort. The frontend ran over its own estimates on Clean
Arch reorg + codegen + reviewer-requested fixes, and a later design-fidelity
pass (matching a Claude Design mockup closely — fonts, exact colors,
page-transition animations) accounts for the biggest chunk. In hindsight,
that pass chased a level of visual polish the brief explicitly says not to
spend time on ("clean and usable beats pretty") — worth flagging honestly
rather than absorbing quietly.
